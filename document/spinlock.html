      🔒
      Spinlock이란?

      Spinlock은 락을 획득하기 전까지 while 루프를 돌며 대기하는 락이다. Context Switching을 막고, 다른 스레드가 락을 해제하는 순간 즉시 선점할 수 있다는 점이 특징이다.

      하나의 Spinlock 인스턴스를 여러 스레드에서 lock_guard로 감싸 공유한다.

      ✓ 장점

      유저 모드에서 계속 대기 가능 → 커널 전환 오버헤드 없음.
      락 경쟁이 짧은 경우 Mutex보다 훨씬 빠르다.

      ✗ 단점

      다른 스레드가 락을 오래 점유하면, 대기 스레드는 CPU를 낭비하며 아무 일도 안 함.
      싱글 코어 환경에서는 데드락에 가까운 상황이 발생할 수 있다.

      💡
      언제 Spinlock을 써야 하나?
      임계 영역이 매우 짧고 (수 μs 이하), 멀티코어 환경일 때. 긴 작업엔 Mutex가 낫다.
      ⚠️
      잘못된 구현 — 왜 동작하지 않는가
       spinlock_broken.cpp
      class Spinlock {
public:
    void lock() {
        while(_locked) {}   // ① 루프 대기
        _locked = true;     // ② 별도 대입 — 원자적이지 않음!
    }
    void unlock() { _locked = false; }

private:
    volatile bool _locked = false;
};

      실제로 일어나는 일 (Race Condition)

      우리가 원하는 흐름과 현실은 다르다.

      🎯 우리가 원하는 것
      1
      t1의 락이 해제 → _locked = false
      2
      t2가 루프를 탈출 → _locked = true
      3
      t1은 t2가 끝날 때까지 대기
      💥 현실
      1
      t1이 unlock → _locked = false
      2
      t2가 루프 탈출 (false 확인)
      3
      t1도 루프 탈출! t2보다 먼저 true 대입 → 락 재획득
      4
      t2가 true 보고 다시 대기 → t1이 연속 2번 락 획득
      5
      결과: t2는 락을 0번 획득, t1은 2번 연속 획득 가능
      ⚡
      핵심 문제: while() 탈출과 = true 대입이 두 개의 별도 명령이라
      그 사이에 다른 스레드가 끼어들 수 있다. 멀티스레딩에서 어셈블리 1줄 단위로 선점이 일어난다.
      ⚛️
      해결책 — CAS (Compare And Swap)

      "루프 탈출"과 "값 변경"을 하나의 원자적 명령으로 합쳐야 한다. C++11이 제공하는 std::atomic의 CAS 연산이 이를 해결한다.

      CAS 동작 원리

      ① 메모리의 실제 값
      false
      현재 락이 해제된 상태
      ② 내가 기대하는 값 (expected)
      false
      두 값이 같으면 → desired로 교체 & true 반환

      두 값이 다르면 → 교체 없이 false 반환, expected에 실제 값을 써넣음.

       spinlock_correct.cpp
      class Spinlock {
public:
    void lock() {
        bool expected = false;
        bool desired  = true;

        while(!_locked.compare_exchange_strong(expected, desired)) {
            expected = false; // expected는 ref → 실패 시 실제 값으로 덮어써짐, 초기화 필수!
        }
        // while 탈출 == CAS 성공 == 이미 원자적으로 desired(true) 세팅 완료
        // store(true) 별도 호출 불필요 — compare_exchange 성공 시 값이 이미 교체됨
    }

    void unlock() {
        _locked.store(false); // Atomic 변수의 안전한 값 할당
    }

private:
    atomic<bool> _locked = false; // CAS를 위한 Atomic 변수
};

      compare_exchange_weak vs strong

      메서드spurious failure성능권장 사용
      compare_exchange_weak⚠️ 있음 (가짜 실패 가능)빠름 (일부 아키텍처)루프 내부 (while 자체가 재시도)
      compare_exchange_strong✅ 없음약간 느림루프 밖, 단발성 체크
      📌
      실전에서는 while 루프 안이므로 weak를 써도 되지만, 명확한 의도 전달을 위해 Spinlock 구현에선 strong이 더 읽기 편하다. Spurious Failure 문서에서 가짜 실패 원리를 확인할 수 있다.
      ❓
      면접 단골 — volatile의 진짜 의미
      언어volatile의 의미멀티스레드 안전성
      C++ 컴파일러에게 "이 변수를 최적화하지 말 것"을 지시.
      변수의 레지스터 캐싱·dead store 제거를 막는다.
      ⚠ 컴파일러의 명령어 재배치(reordering)는 막지 못함      		 ❌ 보장 안 함. CPU 하드웨어 캐시·명령어 재배치는 여전히 발생.
      재배치 방지는 std::atomic의 memory order가 담당.
      C# / Java 멀티스레드 환경에서 가시성(Visibility) 보장.
      여러 스레드에 의해 수정될 수 있음을 명시한다.      		 ⚠️ 원자성 보장은 아님. 가시성만 보장.
      🎤
      예상 면접 질문: "C++에서 volatile bool로 Spinlock을 만들면 안 되는 이유는?"

      모범 답변: volatile은 컴파일러 최적화만 막을 뿐, CPU 수준의 메모리 재배치와 캐시 일관성 문제는 해결하지 못합니다. 스레드 안전성을 위해선 C++11의 std::atomic이나 std::mutex를 사용해야 합니다.
      📊
      동기화 수단 비교 — 면접 정리
      수단대기 방식커널 전환적합한 상황
      Spinlock 바쁜 대기 (busy-wait) ❌ 없음 임계 영역이 매우 짧음 (μs 단위), 멀티코어
      std::mutex 슬립 (OS 스케줄러에 양보) ✅ 있음 임계 영역이 길거나, 대기 시간이 예측 불가
      std::atomic Lock-free 연산 ❌ 없음 단순 정수 카운터, 플래그 변수
      🎤
      추가 예상 질문: "게임 서버에서 Spinlock과 Mutex 중 어느 것을 선택하겠나?"

      모범 답변: 네트워크 패킷 수신 큐처럼 락 경합이 짧은 경우엔 Spinlock, DB 쿼리나 파일 I/O처럼 대기가 긴 경우엔 Mutex를 선택합니다. 프로파일링 결과 없이 무조건 한 쪽을 고집하면 안 됩니다.

      최종 완성 코드 (전체)
       spinlock_final.cpp
      #include <atomic>
#include <thread>
#include <mutex>
#include <iostream>
using namespace std;

class Spinlock {
public:
    void lock() {
        bool expected = false;
        bool desired  = true;
        while (!_locked.compare_exchange_strong(expected, desired))
            expected = false; // ref 변수이므로 매 루프마다 초기화
    }
    void unlock() {
        _locked.store(false);
    }

private:
    atomic<bool> _locked = false;
};

Spinlock spinLock;
int32_t sum = 0;

void Add() {
    for (int i = 0; i < 100'000; ++i) {
        lock_guard<Spinlock> g(spinLock);
        sum++;
    }
}

void Sub() {
    for (int i = 0; i < 100'000; ++i) {
        lock_guard<Spinlock> g(spinLock);
        sum--;
    }
}

int main() {
    thread t1(Add), t2(Sub);
    t1.join(); t2.join();
    cout << "sum: " << sum << endl; // 항상 0
}
