    Level Lv. -

      🧩
      왜 CPU마다 CAS 구현이 다른가

      C++의 std::atomic::compare_exchange는 언어 수준 API일 뿐이다. 실제로는 컴파일러가 대상 CPU의 원자 연산 명령으로 lowering한다.

      아키텍처마다 “한 번에 비교하고 바꾸기”를 구현하는 방식이 다르기 때문에, Spurious Failure 같은 현상이 x86과 ARM에서 다르게 나타난다.

      아키텍처대표 명령CAS 구현 방식spurious failure
      x86 / x86-64 LOCK CMPXCHG 단일 명령, 캐시 라인 잠금 사실상 없음
      ARM / AArch64 LDXR + STXR (LL/SC) 읽기(링크) → 쓰기(조건부) 2단계 있음 (weak 허용)
      🖥️
      x86 — LOCK CMPXCHG

      CMPXCHG (Compare and Exchange)는 메모리 값과 레지스터(expected)를 비교한 뒤, 같으면 desired로 교체하고 ZF=1, 다르면 메모리 값을 레지스터에 넣고 ZF=0을 설정한다.

      앞에 LOCK 접두사가 붙으면, 해당 메모리 주소의 캐시 라인 전체를 배타적으로 잠그고 읽기-비교-쓰기를 한 덩어리로 수행한다. 다른 코어는 그 라인에 접근할 수 없다.

       cmpxchg_pseudo.asm
      ; 의사 코드 — bool CAS(int* mem, int expected, int desired)
LOCK CMPXCHG [mem], desired   ; *mem == EAX(expected) 이면 desired로 교체
; 성공: ZF=1, 실패: ZF=0 & EAX에 *mem의 실제 값 로드
      1
      LOCK 획득: 대상 캐시 라인을 Modified/Exclusive 상태로 독점.
      2
      비교 + 교체: 메모리 값 == expected → desired 쓰기. 아니면 expected 레지스터 갱신.
      3
      LOCK 해제: 단일 명령 종료와 함께 버스/캐시 잠금 해제.
      ✓ 특징

      하나의 원자 명령으로 CAS 완료. 값이 같으면 실패하지 않음 → compare_exchange_strong에 적합.

      ✗ 비용

      캐시 라인 잠금으로 경합 시 다른 코어가 대기. 잠금 범위가 넓어 스케일링에 부담이 될 수 있음.

      📱
      ARM — Load-Linked / Store-Conditional

      ARM은 CMPXCHG 같은 단일 명령 대신 LL/SC (Load-Linked / Store-Conditional) 쌍으로 원자 연산을 만든다. AArch64에서는 LDXR(Load-Exclusive) + STXR(Store-Exclusive)에 해당한다.

      ① Load-Linked (LDXR)
      값 읽기 + 주소 “감시” 시작
      이 주소에 대한 독점 추적(reservation) 등록
      ② Store-Conditional (STXR)
      감시 유효 시에만 쓰기
      성공 0 / 실패 1 반환. 실패해도 메모리 값은 안 바뀔 수 있음
       cas_ll_sc_pseudo.cpp
      bool cas_ll_sc(int* mem, int expected, int desired) {
    int loaded;
    int status;

    do {
        loaded = LDXR(mem);          // Load-Linked: 읽고 reservation 설정
        if (loaded != expected)
            return false;            // 진짜 실패 — 값 불일치

        status = STXR(mem, desired);   // Store-Conditional: reservation 유효할 때만 쓰기
    } while (status != 0);           // SC 실패(가짜 실패 포함) → 재시도

    return true;
}

      Store-Conditional이 실패하는 대표 사유

      사유설명메모리 값 변경?
      다른 코어의 쓰기 감시 중인 캐시 라인에 다른 스레드가 store → reservation 무효화 다른 스레드가 바꿨을 수 있음 (진짜 실패)
      컨텍스트 스위치 / 인터럽트 LL과 SC 사이에 스레드가 선점되면 reservation이 깨질 수 있음 값은 그대로일 수 있음 (가짜 실패)
      LL/SC 사이 명령 과다 ARM은 LL~SC 사이 허용 명령 수에 제한이 있음 (구현·ISA 의존) 값은 그대로일 수 있음 (가짜 실패)
      캐시 라인 경합 같은 라인에 대한 빈번한 invalidation 상황에 따라 다름
      ⚡
      핵심: SC가 실패했다고 해서 항상 “값이 달라졌다”는 뜻은 아니다. reservation만 깨졌을 수 있다. 이것이 Spurious Failure의 하드웨어적 근거이다.

      CMPXCHG vs LL/SC — 비교 정리
      항목x86 LOCK CMPXCHGARM LDXR / STXR
      명령 수 1개 (atomic) 2개 이상 (LL + SC, 실패 시 반복)
      동기화 단위 캐시 라인 LOCK per-address reservation (링크)
      실패 시 값 불일치 필수? ✅ 예 — 값이 다를 때만 실패 ❌ 아님 — reservation 깨짐만으로도 실패
      C++ 매핑 compare_exchange_strong에 가깝게 동작 compare_exchange_weak의 spurious failure 허용과 부합
      게임 서버 맥락 Intel 서버, PC 클라이언트 빌드 모바일(ARM), 일부 ARM 서버, 콘솔
      🔗
      C++에서의 연결 — weak / strong

      C++ 표준은 하드웨어 차이를 흡수하기 위해 두 API를 제공한다. 컴파일러는 같은 소스 코드를 x86에서는 CMPXCHG로, ARM에서는 LDXR/STXR 루프로 생성한다.

       compare_exchange.cpp
      atomic<bool> flag;

// x86: LOCK CMPXCHG 1회, 실패 = 값 불일치만
// ARM: LDXR/STXR 루프, SC 실패 = 가짜 실패 가능
flag.compare_exchange_weak(expected, desired);  // spurious failure 허용
flag.compare_exchange_strong(expected, desired); // 내부 재시도로 spurious 제거
      🎤
      예상 면접 질문: "x86과 ARM에서 CAS 구현이 다른 이유는? 왜 weak에 spurious failure가 있나?"

      모범 답변: x86은 LOCK CMPXCHG로 단일 명령 CAS를 제공하고, ARM은 LL/SC 두 단계로 원자성을 구현한다. SC는 reservation이 깨지면 값이 같아도 실패할 수 있어 C++ weak가 이를 허용한다. strong은 실패 시 내부 재시도로 spurious failure를 제거한다.
