    Spurious Failure

    compare_exchange_weak가 값이 같아도 실패하는 현상 — 가짜 실패의 원인과 대응 방법

    Level Lv. -
    Tags
      ⚛️
      Spurious Failure란?

      Spurious Failure(가짜 실패)는 std::atomic::compare_exchange_weak가 메모리 값이 expected와 일치함에도 불구하고 false를 반환하는 현상이다.

      진짜 실패(값 불일치)와 달리, 다른 스레드가 값을 바꿨기 때문이 아니다. CPU·하드웨어 구현 특성 때문에 CAS 연산 자체가 “그냥 실패”할 수 있다.

      💡
      한 줄 요약: weak CAS는 “값이 달라서”뿐 아니라 “이유 없이”도 실패할 수 있다. 그래서 while 루프로 재시도하는 패턴과 짝을 이룬다.
      🔀
      진짜 실패 vs 가짜 실패
      구분원인expected 갱신재시도
      진짜 실패
      (genuine failure)      		 다른 스레드가 먼저 값을 바꿈.
      메모리 값 ≠ expected      		 ✅ 실제 메모리 값으로 덮어씀 새 expected로 다시 CAS
      가짜 실패
      (spurious failure)      		 LL/SC 계열 명령의 중단, 캐시 라인 경합, 스펙큘레이션 등
      값은 여전히 일치할 수 있음      		 ⚠️ 구현에 따라 달라질 수 있음 동일 expected로 즉시 재시도 가능
      진짜 실패 예시

      락이 false인데 다른 스레드가 먼저 true로 바꿈 → CAS 실패, expected가 true로 갱신됨.

      가짜 실패 예시

      메모리는 여전히 false인데 ARM의 SC(Store-Conditional)가 중단됨 → CAS가 실패 반환. 값은 안 바뀜.

      🔧
      왜 발생하는가 — 하드웨어 관점

      x86의 LOCK CMPXCHG는 단일 명령으로 CAS를 수행해 spurious failure가 사실상 없다. 반면 ARM·RISC-V 등은 Load-Linked / Store-Conditional (LL/SC) 패턴을 쓴다.

      1
      Load-Linked: 주소의 값을 읽고 “이 주소를 감시”한다.
      2
      Store-Conditional: 감시 중 값이 바뀌지 않았으면 새 값을 쓴다. 바뀌었거나 중단되면 실패.
      3
      중단 사유: 같은 캐시 라인에 대한 다른 코어의 쓰기, 인터럽트, 컨텍스트 스위치, LL/SC 사이 너무 많은 명령 등

      C++ 표준은 compare_exchange_weak가 이런 하드웨어 특성을 허용하도록 정의되어 있다. 컴파일러·CPU가 weak를 더 빠르게 구현할 수 있게 하기 위함이다.

      🔁
      weak + while 루프 — 올바른 사용법

      가짜 실패는 재시도하면 해결된다. Spinlock처럼 루프 안에서 CAS를 호출할 때 weak가 적합한 이유다.

       spinlock_weak.cpp
      void lock() {
    bool expected = false;
    bool desired  = true;

    while (!_locked.compare_exchange_weak(expected, desired)) {
        expected = false; // 진짜 실패 시 expected가 true로 바뀔 수 있음 → 초기화
        // 가짜 실패면 expected는 false 그대로 → 즉시 재시도
    }
}
      ⚡
      주의: 루프 밖에서 단 한 번만 CAS 결과를 믿어야 한다면 compare_exchange_strong을 써야 한다. strong은 spurious failure를 허용하지 않는다.
      📊
      weak vs strong — 선택 기준
      메서드spurious failure성능권장 사용
      compare_exchange_weak ⚠️ 있음 (가짜 실패 가능) 빠름 (일부 아키텍처) while 루프 내부 — 실패 시 재시도 전제
      compare_exchange_strong ✅ 없음 약간 느림 루프 밖, 단발성 조건 체크
      🎤
      예상 면접 질문: "weak와 strong의 차이는? Spinlock에 뭘 쓰겠나?"

      모범 답변: weak는 spurious failure가 있어 루프 재시도가 전제다. Spinlock은 어차피 while로 돌리므로 weak도 동작하지만, 의도를 명확히 하려면 strong을 쓰기도 한다. Spinlock & CAS 문서에서 전체 맥락을 확인할 수 있다.
